UNIT-3: JDBC (Java Database Connectivity)

Connecting Java applications to relational databases for robust, data-driven systems.

Introduction to JDBC

JDBC (Java Database Connectivity) is a Java API to connect and execute queries with the database. It is a part of JavaSE (Java Standard Edition).

JDBC API allows Java programs to connect to databases. Database access is the same for all database vendors, as the **JVM uses a JDBC driver to translate generalized JDBC calls into vendor specific database calls**.

Typical JDBC Programming Procedure (5 Steps)

The complete workflow for interacting with a database usually follows these steps:

  1. Load the database driver: Register the Driver class (e.g., Class.forName("oracle.jdbc.driver.OracleDriver")).
  2. Obtain a connection: Establish a secure connection using DriverManager.
  3. Create and execute statements (SQL queries): Prepare SQL commands for execution.
  4. Use result sets (tables) to navigate through the results: Process the data returned by SELECT queries.
  5. Close the connection: Release the database resources.

Types of JDBC Drivers

There are four general types of JDBC drivers:

1. Type-1 Driver (JDBC-ODBC Bridge Driver)

  • Mechanism: Uses an **ODBC driver** to connect to the database. It converts JDBC method calls into ODBC function calls.
  • Note: Type-1 driver is also called **Universal driver** because it can be used to connect to any database that has an ODBC driver.

2. Type-2 Driver (Native-API Driver)

  • Mechanism: Uses the client-side libraries of the database. This driver converts JDBC method calls into **native calls of the database API**.
  • Note: Data transfer is much more secure compared to Type-1, but requires native database libraries on the client machine.

3. Type-3 Driver (Network Protocol Driver)

  • Mechanism: Fully written in Java (portable). Uses **middleware (application server)** that converts JDBC calls into the vendor-specific database protocol.
  • Pros: Highly portable; no need for individual client-side installation of drivers.

4. Type-4 Driver (Thin Driver / Native Protocol Driver)

  • Mechanism: Interacts **directly with the database** using the native protocol. This driver is entirely written in Java.
  • Note: It does not require any native database library, which is why it is known as the **Thin Driver**. It is the most common choice today.

Establishing Connection with Data Base (DriverManager)

DriverManager Class

The purpose of the java.sql.DriverManager class is to provide a common access layer on top of different database drivers. It requires that each driver must be registered before use.

Driver Loading and Connection

This demonstrates the two initial steps (Load Driver and Obtain Connection) for a Type 4 JDBC Driver (Oracle):

// 1. Load the database driver using ClassLoader
Class.forName ("oracle.jdbc.driver.OracleDriver");

// 2. Obtain a Connection
Connection con = DriverManager.getConnection (
  "jdbc:oracle:thin:@bonsai.ite.gmu.edu:1521:ite", // JDBC URL
  "accountname", // Database Username
  "password"); // Database Password

java.sql.* Package: Interfaces and Statements

The java.sql package contains the core classes and interfaces for the JDBC API.

Key Interfaces of java.sql.*

The most important interfaces are:

  • Driver interface
  • Connection interface
  • Statement interface
  • PreparedStatement interface
  • CallableStatement interface
  • ResultSet interface
  • ResultSetMetaData interface
  • DatabaseMetaData interface
  • RowSet interface

1. Statement Interface

The Statement interface provides methods to execute queries with the database. It acts as a factory for ResultSet objects.

  • public ResultSet executeQuery(String sql): Used to execute SELECT queries.
  • public int executeUpdate(String sql): Used to execute DDL (CREATE, DROP, ALTER) or DML (INSERT, UPDATE, DELETE) queries. Returns the number of rows affected.
  • public boolean execute(String sql): Used for queries that may return multiple results.

Creating Tables and Data (DDL/DML)

Statements that create, alter, or drop a table are DDL (Data Definition Language) statements and are executed with executeUpdate().

Statement stmt = con.createStatement();

// Creating a table (DDL)
stmt.executeUpdate (
  "CREATE TABLE COFFEES (COF_NAME VARCHAR(32), SUP_ID INTEGER, " +
  "PRICE FLOAT, SALES INTEGER, TOTAL INTEGER)");

// Inserting data (DML)
stmt.executeUpdate ( "INSERT INTO COFFEES " +
  "VALUES ('Colombian', 101, 7.99, 0, 0)");
stmt.executeUpdate ( "INSERT INTO COFFEES " +
  "VALUES ('French_Roast', 49, 8.99, 0, 0)" );
// ... more inserts

In practice, executeUpdate is used more often to **update** tables than to create them. However, executeQuery is the method used most often for executing **SELECT** statements.

2. PreparedStatement Interface

The PreparedStatement interface is a subinterface of Statement used to execute **parameterized queries**. This uses placeholders (?) for values, which are set using setter methods (e.g., setString(), setFloat()).

String sql="INSERT INTO emp VALUES(?, ?, ?)";
PreparedStatement pstmt = con.prepareStatement(sql);

// Value is set by calling the setter methods
// pstmt.setString(1, "Value for first ?");

pstmt.executeUpdate();

3. CallableStatement Interface

Used to call **stored procedures and functions** from a Java program. Stored procedures are beneficial for complex scenarios and executing logic directly on the database server.

// Calling a stored procedure named 'myprocedure'
CallableStatement stmt = con.prepareCall("{call myprocedure(?,?)}");

ResultSet and ResultSetMetaData

ResultSet Interface

The ResultSet object maintains a cursor pointing to a row of the table returned by a SELECT query. Initially, the cursor points to before the first row. The rs.next() method moves the cursor to the next row.

Getting Data From a Table

ResultSet rs = stmt.executeQuery ("SELECT COF_NAME, PRICE FROM COFFEES");

while (rs.next())
{
  String s = rs.getString ("COF_NAME");
  float n = rs.getFloat ("PRICE");
  System.out.println (s + " " + n);
}

ResultSetMetaData Interface

Metadata means **data about data**. This interface provides methods to get metadata of a table (e.g., total number of columns, column name, column type) from the ResultSet object. It is useful for inspecting database schema information dynamically.

JDBC Data Source Architecture

This architecture represents a more enterprise-level approach, often involving JNDI (Java Naming and Directory Interface) for managing connections centrally, improving performance and scalability.

The flow is typically: **Application** → **JNDI** → **Connection Manager** → **JDBC** → **Database**.
